# Spec: Wer kennt mehr?

## Goal
A game where both teams compete to name *more* of a given thing than the other team (e.g. "Nennt so viele europΟΛische HauptstΟΛdte wie mΟΕglich"). Three **scoring modes** (config `scoringMode`, default `standard`) decide how it scores:

- **`standard`** (default, mid-game behaviour): scores exactly like every other game. No points are awarded per round; instead the gamemaster records who named more each round (Team 1 / Team 2 / Unentschieden) and a running **round-win tally** accumulates as scorekeeping guidance (GM-only βÄî the show frontend stays clean). After the last round a summary reward screen shows the tally and the host awards the **positional game points** (`currentIndex + 1`) to the chosen team(s). Honors the **Aufholjoker** (Ο½2 for the armed team), like every other positional-points game.
- **`count`** (final-game behaviour): the host counts how many valid items each team named; the team that named more wins the round and is awarded **points equal to that count**, inline. Because the count can be large (15+), it rewards big swings in the global score.
- **`count-penalty`** (final-game behaviour, high stakes): identical to `count`, but the **losing team also loses** the entered count (clamped at 0 by the reducer). A tie (both teams selected) **changes nothing** βÄî no points awarded or deducted.

## Acceptance criteria
- [ ] Starts on the **question** phase: question text (+ optional `info` subtitle, optional `questionImage`, optional `timer`) is shown; teams call out their answers live
- [ ] A host advance (nav-forward) reveals the **answer** phase: the examples are shown βÄî a single string (`answer`) in the standard `.quiz-answer` box, or an **auto-fitting grid** (`answerList`) that always uses the full width of the answer box. A handful of items stretch to fill the row; a long list wraps into as many columns as fit and flows downward. Columns collapse to a single full-width column on narrow screens; nothing ever spills outside the box
- [ ] **`count` mode:** the answer phase shows a scoring host panel: two **toggle** team buttons (Team 1 / Team 2) + a number input ("Anzahl") + a "Punkte vergeben" button. Selecting one team = that team wins βÜ£ `onAwardPoints(team, count)`. Selecting **both** teams = tie βÜ£ each team gets `floor(count / 2)`. The button is disabled until at least one team is selected; submitting awards points and advances to the next question (or completes the game)
- [ ] **`count-penalty` mode:** reuses the exact same `count`-mode host panel + gamemaster controls. Selecting one team = that team wins βÜ£ `onAwardPoints(winner, count)` **and** `onAwardPoints(loser, -count)` (the reducer clamps the loser at 0). Selecting **both** teams = tie βÜ£ **no points change** (no split, no penalty); the tie hint reads "Unentschieden βÄî keine PunkteΟΛnderung." instead of the count-mode split hint
- [ ] **`standard` mode:** no points are awarded per round and the **show frontend stays clean** (no on-show scoring panel βÄî each round is question βÜ£ reveal examples βÜ£ **nav-forward advances**). On the gamemaster, the answer phase of each **real** round (index 0 example excluded) exposes a **round-win recorder**: a `round-winner` button-group (`round-team1` / `round-team2` / `round-draw`, labelled Team 1 / Team 2 / Unentschieden; tapping the already-active button clears it) plus an `info` control showing the running **round-win tally** (`Rundenstand βÄî {t1}: X ¬Ζ {t2}: Y`, and `¬Ζ Unentschieden: Z` when any draws). The selection is per-round (keyed by question index, so back-nav reveals it) and **optional** βÄî nav-forward advances regardless. After the last question the game enters a **summary** reward screen with three buttons (Team 1 / Team 2 / Unentschieden), shown on the frontend like any other game's points screen (with the tally as guidance when at least one round was recorded) and mirrored on the gamemaster (`final-team1` / `final-team2` / `final-draw`, plus the `info` tally). Picking a winner awards `pointValue` (`currentIndex + 1`) to the chosen team(s) (Unentschieden βÜ£ both) and completes the game. The **host** still decides the overall winner βÄî the tally is guidance only, never auto-applied
- [ ] **`standard` mode honors the Aufholjoker:** the summary reward awards `pointValue * 2` to the Aufholjoker-armed team (the trailing team that spent the comeback joker), mirroring `BaseGameWrapper.handleComplete`; the armed flag is then cleared by the wrapper's inline-scored `onGameComplete` branch. On a draw only the armed team's points double. The armed team's summary award button shows the same "Ο½2 Aufholjoker" `award-double-badge` as the `AwardPoints` screen. (`count` / `count-penalty` modes remain out of scope βÄî see [comeback-joker.md](../comeback-joker.md))
- [ ] `scoringMode` defaults to `standard` (the normal positional point system). Selected game-wide in the admin GameEditor base settings via a **"Punktevergabe" `<select>`** (shared `.be-scoring-mode` class, `min-width: 0` so it never overflows on narrow screens) whose options are, in order: "Standard" (βÜ£ `standard`, the default, **stored as `undefined`**), "Trefferzahl" (βÜ£ `count`), "Trefferzahl mit Abzug" (βÜ£ `count-penalty`). Labels are short so the rules footer fits one line; the full meaning is a `title` tooltip on each option
- [ ] On answer reveal the page anchors to the **answer** (`useQuizAutoScroll(..., 'answer')`) βÄî the same target as the gamemaster **Antwort** jump-button (`scrollShowToAnchor('answer')`): the `.quiz-answer` examples grid is brought just below the sticky header and leads the viewport (the question scrolls above the header), not the scoring panel below it
- [ ] In **count** mode, once the host **starts scoring** the round βÄî selecting a team or editing the count, on the frontend OR via the gamemaster (`toggle-team1/2`, `award-submit:change` commands; the on-card team buttons; the **Anzahl** input's `onFocus`) βÄî a `scoringActive` flag flips the answer-phase anchor to the **bottom** so the on-show `.bet-quiz-host-panel` stays in view and the projector follows what the host enters, even as the tie hint grows the card. `scoringActive` resets whenever the phase is not `answer`, so the next reveal (or a back-nav re-reveal) leads with the answer again. (In **standard** mode there are no **on-show** per-round controls βÄî the round-win recorder is gamemaster-only βÄî so the answer always stays anchored.)
- [ ] The **summary** phase keeps the **bottom** anchor (no answer to show βÄî surface the final scoring controls instead)
- [ ] Optional per-question `timer` counts down exactly like `simple-quiz` (shared `Timer`, fixed bottom-left, GM Stop/Pause aware)
- [ ] After the last question, the next advance calls `onGameComplete()`
- [ ] Uses `BaseGameWrapper` with `requiresPoints` + `skipPointsScreen` + `hideCorrectTracker` (points awarded inline, no `AwardPoints` screen)
- [ ] Question index 0 is a non-scoring **"Beispiel Frage"** practice round (its scoring panel advances without awarding points); real questions are labelled `Frage N von M` (M = `questions.length - 1`)
- [ ] `randomizeQuestions` / `questionLimit` honoured via `useShuffledQuestions`; `disabled` questions filtered
- [ ] Gamemaster controls wired: in `count` mode the answer phase mirrors the on-screen panel (toggle team buttons + number input + submit). In `standard` mode the answer phase exposes the GM-only round-win recorder (`round-winner` button-group `round-team1/2/draw` + an `info` round-win tally; nav-forward still advances); the summary phase exposes a `final-winner` button-group (`final-team1` / `final-team2` / `final-draw`) alongside the `info` tally βÄî the `round-*` and `final-*` ids are all distinct from the wrapper's `award-team1/2/draw` so `BaseGameWrapper`'s command listener doesn't intercept them. A selected team toggle (count mode) is clearly marked in the GM by a near-white `outline` ring (`.gm-btn--active`) βÄî using `outline` rather than box-shadow/border so it can't be defeated by the higher-specificity `.gm-btn:hover` reset or by sticky `:hover` on touch (both teams selected = two rings at once, shown instantly). The examples are published via the structured `answerList` field (`{rank, text, revealed: true}[]`) so the GM renders them once as the ranking-style pill grid (`.gamemaster-answer-list`) βÄî not duplicated as a plain `answer` blob plus an `extraInfo` list
- [ ] **Selectable in the admin**: the type appears in the "Neues Spiel erstellen" modal (`GamesTab` `GAME_TYPES`) and the `GameEditor` type `<select>`. These arrays are plain `GameType[]` literals βÄî **not** type-checked for completeness βÄî so a new type compiles fine yet is invisible in the admin until added to both. (A `GAME_TYPE_INFO` label + `GAME_TYPE_TEMPLATES` entry are `Record<GameType, βÄΠ>` and therefore enforced by `tsc`.)

## State / data changes
- No `AppState` changes
- Config type: `WerKenntMehrConfig` (`type: 'wer-kennt-mehr'`) in [`src/types/config.ts`](../../src/types/config.ts)
  - `scoringMode?: 'count' | 'standard' | 'count-penalty'` βÄî game-level scoring mode (default `standard`; `undefined` β΅£ `standard`). Validated in `validate-config.ts` (only on this type, only those three values). Schema in [`specs/api/openapi.yaml`](../api/openapi.yaml) `WerKenntMehrConfig`
- Question type: `WerKenntMehrQuestion`:
  - `question: string` βÄî the prompt
  - `info?: string` βÄî optional subtitle shown above the question
  - `questionImage?: string` βÄî optional question image (raw logical path; encoded at the DOM boundary). **No answer image.**
  - `answer?: string` βÄî single example answer (used when no list)
  - `answerList?: string[]` βÄî list of example answers, rendered as the compact grid. At least one of `answer` / non-empty `answerList` required
  - `timer?: number` βÄî optional time limit in seconds
  - `disabled?: boolean`
- Points awarded via `onAwardPoints(team, n)` (the standard `AWARD_POINTS` action; balance clamped at 0). In `count` mode `n` is the entered count per round; in `count-penalty` mode the winner gets `+count` and the loser `-count` (clamped at 0), tie = no calls; in `standard` mode `n` is `pointValue` (`currentIndex + 1`) βÄî or `pointValue * 2` for the Aufholjoker-armed team βÄî awarded once on the summary reward screen. `BaseGameWrapper` keeps `skipPointsScreen` in all modes βÄî `standard` mode renders its own summary/reward screen inside the game (so it sits on the light game card with legible on-card text), and its `finishGame` applies the Aufholjoker Ο½2 itself (the wrapper's `handleComplete` multiplier is never reached under `skipPointsScreen`)

## UI behaviour
- Component: [`src/components/games/WerKenntMehr.tsx`](../../src/components/games/WerKenntMehr.tsx)
- Phase machine: `question βÜ£ answer` (`count` mode), or `question βÜ£ answer βÜ£ βÄΠ βÜ£ summary` (`standard` mode, summary after the last round). Reuses [`QuizQuestionView`](../../src/components/games/QuizQuestionView.tsx) for the question render (info, emoji scaling, `questionImage` via `RetryImage` + lightbox, `Timer` portal) with `showAnswer={false}`; the examples are rendered by this component on reveal
- Single-string `answer` reuses the `.quiz-answer` box; `answerList` renders into a new `.wkm-examples` CSS grid: `grid-template-columns: repeat(auto-fit, minmax(min(100%, 200px), 1fr))`. `auto-fit` + `1fr` makes the present items stretch to fill the full box width whatever the count (few items spread across the row as wide pills; long lists wrap into a full multi-column block); `min(100%, 200px)` collapses to a single full-width column on narrow screens. Each item is a **chip**: centered text in a rounded (`12px`), success-tinted card (`background rgba(--success-rgb, 0.12)`, `1px` success border). The global `li` `padding`/`border-bottom` (base.css) is reset so no separator dangles. Font is `clamp(1.05rem, 1.9vw, 1.6rem)` for projector readability. Long answers (e.g. a German compound like `Mandragoren-Wiederbelebungstrank (Mandrake Restorative Draught)`) wrap **inside** the chip via `overflow-wrap: anywhere` + `hyphens: auto` βÄî `anywhere` (not `break-word`) is required because only it shrinks the chip's intrinsic min-content width, so the text never spills over neighbouring chips
- **`count`** mode renders a scoring panel on the answer phase (reuses `.bet-quiz-host-panel` / `.bet-quiz-host-row`; team buttons are independent toggles, both-on = tie). **`standard`** mode renders **no** on-show scoring panel βÄî the answer phase shows only the examples and nav-forward advances; the per-round round-win recorder and running tally live only on the gamemaster. The standard-mode **summary reward screen** renders on the frontend as on-card content (`<h2>Punkte vergeben</h2>` + `.bet-quiz-host-hint` prompt + `.award-points-teams` buttons) βÄî not a nested `#awardPointsContainer`, whose text colour assumes the dark page background and would be illegible on the light game card βÄî and additionally shows the round-win tally (`.wkm-tally`, reusing `.bet-quiz-host-hint` styling) as guidance when at least one round was recorded
- Back navigation: answer βÜ£ question; question βÜ£ previous question (shown revealed), mirroring bet-quiz minus audio
- Backend form: [`src/components/backend/questions/WerKenntMehrForm.tsx`](../../src/components/backend/questions/WerKenntMehrForm.tsx). Per-question fields: `question`, optional `info`, optional `questionImage` (`AssetField`), an examples editor (single `answer` field + an `answerList` free-text area, one entry per line), and `timer`. The `answerList` editor edits a raw multi-line **draft** and only normalizes (split on newline βÜ£ trim βÜ£ drop empties) **on blur**, so the user can type spaces and press Enter to add rows (normalizing per keystroke would strip the new line instantly). Question-level drag-reorder via `useDragReorder`; cross-instance move via `MoveQuestionButton`; ghost-row pattern

## Out of scope
- Per-question or per-answer audio
- Answer image
- Automatic counting / per-team device input of individual items (host enters the final count)
- Validation against any real-world ground truth βÄî authors define the example list in JSON
